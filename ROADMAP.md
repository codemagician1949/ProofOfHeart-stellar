# ProofOfHeart-Stellar — Forensic Audit & Roadmap

> This document was produced as an external technical audit on 2026-05-30. It captures the state of `main` at commit `a9d0c3e` (PR #399). The audit covers architecture, dependencies, bugs, performance, security, technical debt, and a 4-phase roadmap to a production-ready state. The companion GitHub issues file the concrete work items.

## Executive Summary

**Current health: 🔴 BROKEN.** `main` does not compile. `cargo check` fails on a duplicate enum discriminant (`src/errors.rs`) introduced in the last merged PR (#399, `a9d0c3e`), and there is an unreachable `return` in `update_platform_fee` (`src/lib.rs:1379`). Local `cargo test` will fail before producing artifacts. CI did not catch this because the workflow that auto-merged it (`.github/workflows/auto-review.yml`) uses `gh pr merge --admin`, which bypasses branch protection.

Beyond the broken build, the codebase shows the classic fingerprint of an open-source repo that has accepted dozens of small, autogenerated PRs from hackathon contributors without strong review:

- Internally inconsistent state: two error variants share a discriminant, fee validation runs twice with mutually unreachable branches, event payloads disagree with `EVENT_PAYLOADS.md`, `IMPLEMENTATION_SUMMARY.md` falsely claims CI runs `fmt`/`clippy` when it does not, `CONTRIBUTING.md` makes the same claim, and `docs/AUTHORIZATION.md` documents a `update_admin(admin, new_admin)` signature that no longer exists.
- Project-management metadata is checked into source: `fix.md` (excluded by `.gitignore` after being committed), `PR_BODY.md`, `IMPLEMENTATION_SUMMARY.md`, `SECURITY_ASSESSMENT_133.md`. These are workspace artifacts, not artifacts of the contract.
- Two parallel test trees (`src/test.rs` at 4,132 lines + a fragmented `src/tests/` module set) duplicate scaffolding and make coverage gaps invisible.
- Auto-review workflow leaks a different brand identity ("Lernza, a hackathon project. CI has been disabled to maximise velocity"), uses `pull_request_target` with `contents: write`, and auto-merges with an admin PAT — a supply-chain anti-pattern for a smart contract holding user funds.

The vision (transparent, permissionless community-validated crowdfunding) is sound and the core mechanics (campaign lifecycle, voting verification, refund/revenue paths) are implemented. But the codebase is **not production-ready** for any contract that will custody real value. With ~2–4 weeks of focused work, the contract can reach a credible audit-ready state. The recommendations below assume that goal.

## Phase 1 — Deep Codebase Comprehension

### 1.1 Architectural Map

**Single crate, single contract, Soroban WASM target.** Library crate (`crate-type = ["cdylib"]`), `no_std`, panic=abort, LTO on. All logic lives under `src/`:

```
src/
├── lib.rs            (2,216 LOC)  — All public contract entry-points + helpers
├── errors.rs         (88  LOC)    — Error enum (#[contracterror])
├── storage.rs        (901 LOC)    — DataKey enum + every persistent/instance getter+setter
├── types.rs          (130 LOC)    — Campaign, CampaignReserve, Category, PlatformStats, CreateCampaignParams, MaybePendingCreator
├── voting.rs         (181 LOC)    — cast_vote, admin_verify, verify_with_votes, set_params
├── test.rs           (4,132 LOC)  — Monolithic legacy tests
├── tests/            (mod tree)   — Newer split tests (helpers + ~20 modules)
├── *_test.rs         (several)    — Stragglers (admin_transfer_test, benchmark_test, …)
└── *_proptest.rs     (3 files)    — proptest-based property tests
```

**Layering:** `lib.rs` orchestrates. `voting.rs` is the only domain-split helper. `storage.rs` is a thin abstraction over `env.storage().{instance, persistent}` with TTL bumping baked into each setter. There is **no service layer**, no fee calculator, no validator module — all business rules are inlined into the `#[contractimpl]` block.

**Data flow:**

1. **Admin lifecycle**: `init` → (`update_admin` / two-step `initiate_admin_transfer` + `accept_admin_transfer`).
2. **Campaign lifecycle**: `create_campaign` (pending) → `verify_campaign` (admin) **or** `vote_on_campaign × N` → `verify_campaign_with_votes` (community). Once verified, `contribute × N` → either `withdraw_funds` (goal met) or `claim_refund × N` (cancelled / deadline + goal not met).
3. **Revenue lifecycle** (only `Category::EducationalStartup`): after `withdraw_funds`, creator may `deposit_revenue × N`; contributors `claim_revenue`; creator `claim_creator_revenue`.
4. **Vesting (optional)**: `withdraw_funds` splits payout into immediate creator cut and a held reserve; `withdraw_reserve` releases after the configured delay.

**State management:** Two storage planes:
- **`instance`** (cheap, single key) — admin, token, fees, campaign count, voting params, global counters (active/verified/cancelled campaign counts), pause flags, category duration caps.
- **`persistent`** (per-entity, TTL-bumped) — `Campaign(id)`, `Contribution(id, addr)`, `LifetimeContribution(id, addr)`, `HasVoted(id, addr)`, vote tallies/weights, `RevenuePool(id)`, `RevenueClaimed(id, addr)`, `CreatorRevenueClaimed(id)`, `CampaignReserve(id)`, `PersonalCap(id, addr)`, `CampaignStartTime(id)`, `CategoryCampaignsBucket(cat, idx)`, `CreatorCampaignsBucket(addr, idx)`, `ContributorCount(id)`.
- **TTL policy**: writes extend TTL; reads do not (documented in `docs/STORAGE_TTL_POLICY.md`). Constants: bump threshold 7d, bump amount 400d (in ledger units).

**Indexing strategy:** Two bucketed indexes — by category and by creator — sized at 500 IDs/bucket. Sequential ID iteration is used for `list_campaigns`, `list_active_campaigns` (capped at a 200-id scan window), and `get_platform_stats` (capped at 1000). Bucketing is necessary because Soroban's `Vec<u32>` storage entries get expensive to serialize beyond a few hundred entries.

### 1.2 Dependency Analysis

```toml
[dependencies]
soroban-sdk = "=20.1.0"        # PINNED — old (current major is 22.x)

[dev-dependencies]
derive_arbitrary = "=1.3.2"
proptest = "1"
soroban-sdk = { version = "20.1.0", features = ["testutils"] }
```

**Findings:**
- The dependency tree is healthy in *size* — only one runtime dep — but **`soroban-sdk = "=20.1.0"`** is two major versions behind current. Newer SDKs ship storage-fee improvements, `try_invoke_contract` ergonomics, and SEP-41 helpers. The pin is *strict* (`=`), so any contributor running `cargo update` gets nothing. There is no Renovate / Dependabot config to surface updates.
- `Cargo.lock` is committed (correct for a `cdylib` deployed to a network — reproducible WASM hashes).
- `rust-toolchain.toml` exists but is only 68 bytes — likely just pins a stable channel. Worth verifying the channel matches the version `soroban-sdk@20.1.0` was tested against.
- `proptest = "1"` is unpinned (good — patch versions are non-breaking).
- `derive_arbitrary = "=1.3.2"` is pinned exactly. There is no `arbitrary` crate alongside it; verify it's actually used (likely only in `*_proptest.rs`).
- **Missing**: `cargo-audit`, `cargo-deny`, or `cargo-geiger` invocations. A smart contract should be scanning for advisories at every release.

## Phase 2 — Forensic Audit & Gap Analysis

### 2.1 Bugs & Broken Logic (ordered by severity)

#### 🔴 P0 — Build is broken on `main`

**`src/errors.rs:85,87`** — Two enum variants share discriminant `39`:
```rust
InvalidPlatformFee = 39,
TransferAlreadyPending = 39,   // <- E0081
```
`cargo check` confirms: `error[E0081]: discriminant value '39' assigned more than once`. The contract does not build. Fix: bump `TransferAlreadyPending` to `40`. This is the smoking gun for the auto-merge problem.

#### 🔴 P0 — Dead `return` masking the intended error

**`src/lib.rs:1377-1380`**:
```rust
if new_fee > PLATFORM_FEE_MAX_BPS {
    return Err(Error::InvalidPlatformFee);
    return Err(Error::ValidationFailed);   // <- unreachable, warning emitted
}
```
The second `return` is dead code (Rust's `unreachable_code` lint). The first is fine; delete the second line.

#### 🔴 P0 — Init validates the platform fee twice with conflicting errors

**`src/lib.rs:149-151` and `src/lib.rs:164-166`**:
```rust
if platform_fee > PLATFORM_FEE_MAX_BPS {
    return Err(Error::ValidationFailed);   // first check fires
}
// ... token validation ...
if platform_fee > PLATFORM_FEE_MAX_BPS {
    return Err(Error::InvalidPlatformFee); // never reached
}
```
The error semantics intended by `Error::InvalidPlatformFee` are masked by the legacy `Error::ValidationFailed` branch. Keep only the second check.

#### 🟠 P1 — `accept_token_update` strands existing balances

**`src/lib.rs:1338-1352`** changes the token the contract uses for `transfer` without draining or freezing the old token's escrow:

```rust
pub fn accept_token_update(env: Env, admin: Address) -> Result<(), Error> {
    // ...
    let old_token = get_token(&env);
    set_token(&env, &new_token);
    // No migration of escrowed balances; no halt of active campaigns
}
```
After token switch:
- `claim_refund` (line 795) uses `Self::token_client(&env)` which reads the *current* token → tries to refund the new token, but the escrow is in the old token. Transfer either fails or worse, succeeds against any new-token balance the contract happens to hold.
- All `amount_raised` figures are now in mixed denominations.

This is fund-loss-territory unless gated. Minimum mitigation: refuse `accept_token_update` while any non-terminal campaign exists, and force a drain step.

#### 🟠 P1 — Unchecked arithmetic in `contribute` anomaly check

**`src/lib.rs:394`**:
```rust
if amount * 10000 > campaign.funding_goal * AUTO_PAUSE_SINGLE_CONTRIBUTION_BPS_THRESHOLD {
```
Neither multiplication is checked. With `overflow-checks = true` in release, an attacker who can pass `amount` close to `i128::MAX / 10000` will trigger a panic in `contribute` rather than a clean `Error::Overflow`. A panic burns more gas, produces a worse error, and trips Soroban's diagnostic path. Wrap in `checked_mul`.

#### 🟠 P1 — `withdraw_reserve` allows expired-delay claim on a cancelled-ish state without explicit guard

**`src/lib.rs:548-588`** loads the reserve, then loads the campaign, then calls `campaign.creator.require_auth()`. There is **no** check that the campaign is not cancelled. The flow that would put a reserve in place (`withdraw_funds`) requires `!is_cancelled`, and `cancel_campaign` cannot run after `withdraw_funds`, so reaching this state requires a code-path bug or a migration. Still — defense in depth says verify `!campaign.is_cancelled` and `campaign.funds_withdrawn`.

#### 🟠 P1 — `claim_creator_revenue` uses unchecked arithmetic, inconsistent with `claim_revenue`

**`src/lib.rs:925-927`**:
```rust
let contributor_pool = (total_pool * (campaign.revenue_share_percentage as i128)) / 10000;
let creator_share_total = total_pool - contributor_pool;
```
`claim_revenue` (line 879-884) goes through `checked_mul` / `checked_div`. `claim_creator_revenue` does not. Treat both the same way.

Additionally, **line 921**: `if campaign.revenue_share_percentage > 10000` — the constant is `REVENUE_SHARE_MAX_BPS = 5000`. The check threshold should match the create-time validation, or document why it is intentionally looser.

#### 🟠 P1 — `cancel_campaign` does not zero `effective_amount_raised`

**`src/lib.rs:626-662`** marks the campaign cancelled but leaves `effective_amount_raised` at its pre-cancel value. Subsequent `claim_refund` calls decrement it per-contributor. If not every contributor claims, the value drifts. Cosmetic, but breaks any indexer that uses `effective_amount_raised` as a "live contributions" gauge for cancelled campaigns.

Also: `cancel_campaign` reduces `active_campaign_count` (good) and increments `cancelled_campaign_count` (good), but does **not** decrement `total_raised_global` by the amount refundable. The global stays stale until every contributor individually refunds (lib.rs:786-793).

#### 🟠 P1 — `accept_admin_transfer` calls `require_not_paused` *after* `pending_admin.require_auth()`

**`src/lib.rs:1600-1614`** asks the pending admin to sign even when the contract is paused. Re-order so paused contracts reject without consuming signatures (also avoids confusing UX where the user signs and then sees `ContractPaused`).

#### 🟡 P2 — `initiate_admin_transfer` silently overwrites pending admin

**`src/lib.rs:1574-1597`** — when a transfer is already pending, the new call emits a `admin_transfer_cancelled` event for the old pending and overwrites. Campaign transfers (`initiate_campaign_transfer`, line 1979) explicitly reject this with `TransferAlreadyPending`. Inconsistent. Either both should overwrite (and emit cancel events) or both should reject.

#### 🟡 P2 — `set_voting_params` accepts `min_votes_quorum < MAX_VOTES_QUORUM` but voting still allows reaching that quorum easily by Sybil

Acknowledged in `docs/THREAT_MODEL.md`. Mitigation knob (`min_voting_balance`) exists but defaults to 0. Document a sane production default or expose a validator that rejects deployment with `min_voting_balance == 0 && approval_threshold_bps < 7500`.

#### 🟡 P2 — `verify_with_votes` (`src/voting.rs:147-157`) uses unchecked addition

```rust
let total_votes = approve_votes + reject_votes;       // u32 + u32, unchecked
let total_weight = approve_weight + reject_weight;    // i128 + i128, unchecked
```
With `MAX_VOTES_QUORUM = 1000`, `total_votes` overflow is impractical. `total_weight` overflow requires extreme balances but is theoretically possible. Use `checked_add` for consistency with the rest of the file (`cast_vote` already does).

#### 🟡 P2 — `cast_vote` (`src/voting.rs:81,87`) increments `ApproveVotes`/`RejectVotes` without `checked_add` on the u32

Practically unreachable, but inconsistent with the weight path immediately below. Cleanliness.

#### 🟡 P2 — `BlockContributionCount` (global) is dead storage

`get_block_contribution_count` / `set_block_contribution_count` exist (`src/storage.rs:704-716`) but no caller in `lib.rs` writes or reads them — `contribute` uses only the per-campaign counter (`BlockCampaignContributionCount`). Dead code; the `DataKey::BlockContributionCount` variant should be removed unless intentionally reserved for migration.

#### 🟡 P2 — `resume_campaign` couples a contract-level state change to a campaign id

**`src/lib.rs:2157-2186`** requires `require_active_campaign(&campaign)` to lift the **global** `AutoPaused` flag. If the campaign whose contribution triggered the auto-pause has since been cancelled (legitimate), the only way to clear `AutoPaused` is `unpause()` via admin — but `unpause()` only clears `Paused`, not `AutoPaused`. (Verify: line 1050 — yes, `unpause` sets `Paused` to `false`, leaves `AutoPaused`.) **The contract can get stuck auto-paused.**

#### 🟡 P2 — `update_campaign` lifecycle ambiguity

**`src/lib.rs:670-705`** allows editing title+description if `amount_raised == 0`. But there is no check on `is_verified`. So a campaign can be admin-verified at `amount_raised == 0`, then have its title/description swapped *after verification*. This is a "bait-and-switch after admin approval" vector — though gated by no contributions yet, it still lets a creator change *what was verified*. Either block edits post-verification, or document explicitly that verification only attests to the on-chain state at one moment.

The CHANGELOG (#166 fix) mentions this was hardened for `update_campaign_description` once contributions exist, but `update_campaign` itself has the same hole on the verified-but-uncontributed window.

#### 🟡 P2 — `verify_campaigns` (batch) silently skips failed entries past the first

**`src/lib.rs:1131-1176`** captures only `first_error` and continues. Calls beyond a partial-success batch return `Err(first_error)` along with a `campaigns_bulk_verified` event reporting `(verified_count, total_count)`. Callers cannot distinguish "first failed, rest succeeded" from "all failed". Either return `(verified, errors_vec)` or stop on first error.

### 2.2 Performance Bottlenecks

#### `get_platform_stats` is O(MAX_SCAN_LIMIT) on every call

**`src/lib.rs:1904-1940`** scans up to 1,000 campaigns per call. There are already O(1) counters (`ActiveCampaignCount`, `VerifiedCampaignCount`, `CancelledCampaignCount`) maintained by the create/cancel/verify paths. The function should return those counters directly and drop the scan entirely. The `stats_are_partial` flag becomes unnecessary.

#### `accept_campaign_transfer` does a linear bucket search

**`src/lib.rs:2022-2029`** loops over every bucket of the old creator (`0..old_num_buckets`) to find the campaign ID. With the bucket size of 500 and a power user owning thousands of campaigns, this is `O(n)` worst case. Acceptable for now, but indexing creator→bucket position per campaign would make it O(1).

#### `set_campaign_count` is called *after* `set_campaign(&env, count, …)` in `create_campaign`

**`src/lib.rs:283, 311-313`**:
```rust
let mut count = get_campaign_count(&env);
count += 1;
// … sets Campaign(count) first …
set_campaign_count(&env, count);
```
The first `set_campaign(...)` writes to persistent (cost: 1 write), then `set_campaign_count` writes to instance. Fine. The micro-issue is that there is no `checked_add(1)`; if the contract ever reaches `u32::MAX` campaigns, this wraps to 0 and starts overwriting campaign #0. Not a near-term risk, but trivial to make safe.

#### `list_active_campaigns` re-reads each campaign struct

For a busy contract, this is the dominant cost. A future optimisation is to maintain a per-status index (e.g., `ActiveCampaignBucket(idx)` containing IDs of campaigns whose `is_active` flag flipped true). Significant lift; track as roadmap item, not P0.

### 2.3 Security Vulnerabilities

#### 🔴 Auto-merge workflow uses an admin PAT to bypass branch protection

**`.github/workflows/auto-review.yml`** runs on `pull_request_target`, has `contents: write` and `pull-requests: write`, and at line 383 executes:
```yaml
gh pr merge "$PR_NUM" --squash --admin --subject "$PR_TITLE" --body "$MERGE_BODY"
```
using `secrets.OWNER_PAT`. The decision to merge is made by Groq's `openai/gpt-oss-120b` with a system prompt that instructs it to APPROVE almost everything (the prompt is shown verbatim in the workflow). Any contributor whose PR scores `confidence >= 0.7` gets auto-merged with admin override. **This is how the broken-build PR landed.** Mitigations:
1. Remove the `--admin` flag. Branch protection should always win.
2. Stop using `pull_request_target` for arbitrary contributor input. If a fork PR can influence the prompt content (via PR title/body which feeds the model), it can manipulate the verdict.
3. Pin `actions/checkout@v6` to a SHA, not a tag.
4. Either drop auto-merge entirely or require `actions/checkout` + tests to pass first as a separate required check.

#### 🟠 Brand drift in workflow ("Lernza", "Dopey <hello@sshdopey.com>")

The auto-review workflow's prompt opens with `"You are an EXTREMELY LENIENT code reviewer for Lernza, a hackathon project. Interns are shipping fast. CI has been disabled to maximise velocity."` and the merge commit appends `Co-authored-by: Dopey <hello@sshdopey.com>`. This file appears to have been imported from a hackathon project and not adapted. Aside from the optics for a contract that custodies funds, it's a confidentiality and attribution issue: every auto-merged PR commit message advertises "Dopey <hello@sshdopey.com>" as a co-author.

#### 🟠 CI does not run `fmt --check` or `clippy`

**`.github/workflows/ci.yml`**:
```yaml
- run: cargo test --features testutils
- run: cargo build --target wasm32-unknown-unknown --release
```
That's it. Both `CONTRIBUTING.md` line 60-63 and `IMPLEMENTATION_SUMMARY.md` claim these checks run on every PR. They don't. The result: this exact audit found a `unreachable_code` warning that should have failed `cargo clippy -- -D warnings`. Adding fmt and clippy to CI would have caught two of the P0 issues above.

#### 🟠 Admin is a single key — no multisig, no timelock

`update_platform_fee`, `set_campaign_fee_override`, `set_min_voting_balance`, `set_voting_params`, `pause`, `set_creation_disabled`, `set_vesting_params`, `verify_campaign`, `verify_campaigns` (batch), `migrate`, `propose_token_update`, `purge_voting_state` — all gated solely by `assert_admin`. If the admin key is compromised, every campaign on the platform is at risk (admin can force-verify a malicious campaign, set its fee_override to 0, then collude with the creator to drain). Mitigations: timelock on parameter changes, multisig wrapper via an external account contract, or a separate `pauser` role distinct from `admin`.

#### 🟡 Token-weighted voting is Sybil-vulnerable by design

Acknowledged in `docs/THREAT_MODEL.md`. The mitigation (`min_voting_balance`) defaults to 0. Production deployment guides should ship with a non-zero default.

#### 🟡 `verify_campaign_with_votes` is permissionless

By design (`docs/AUTHORIZATION.md` line 25). Anyone can trigger the verification flip once thresholds are met. Acceptable, but should be documented in `EVENT_PAYLOADS.md` so indexers know to filter the `campaign_verified` event payload shape to distinguish admin vs. community paths. (Currently the only distinguishing signal is whether the data payload is empty or contains `approve_votes`.)

#### 🟡 No reentrancy lock on revenue/refund paths

Soroban's execution model is single-threaded per transaction and `token::Client::transfer` does not yield control back to user code (token contract is the trusted SEP-41 surface). Still, the project sets a "checks-effects-interactions" precedent in `withdraw_funds` (line 493-507) but explicitly inverts it in `claim_revenue` (line 894-899) and `claim_creator_revenue` (line 938-946): tokens are transferred **before** state is updated. Comment justifies it as preventing "balance wipe on failed transfer", but this means a malicious token contract (if `accept_token_update` set the contract to a hostile token) could re-enter and double-claim. Pick one pattern and stick to it. If you trust the SEP-41 token, do effects-then-interactions for all flows. If you don't, harden every path.

#### 🟡 `set_personal_cap` does not clamp against existing `lifetime_contribution`

**`src/lib.rs:1533-1555`** lets a contributor set their personal cap to a value below what they have *already contributed*. The next `contribute` call would fail with `ContributionCapExceeded`, which is fine, but it leaves the storage in a confusing state where `lifetime_contribution > personal_cap`. Minor — either reject the lower-than-current set, or document that it freezes further contributions.

### 2.5 Vision Delta Gaps

- **No "challenge" mechanism**: README promises "anyone can challenge a cause" but there is no down-vote-after-verification, no dispute window, no slash. Once verified, a campaign is unchallengeable until the deadline passes.
- **No on-chain dispute resolution**: cancellation is creator-only; admin cannot force-cancel a fraudulent verified campaign. (Admin can `pause`, but that pauses everything.)
- **No frontend repository present in this workspace**: the README references `ProofOfHeart-frontend` — out of scope for this audit, but it likely needs equal scrutiny.

## Closing Notes

Two themes dominate this codebase:

1. **Velocity has outpaced rigor.** The PR throughput is impressive (171 commits in one upstream push), but the auto-merge pipeline, the false documentation claims, and the broken build prove that the safety nets are thinner than they appear. The single highest-leverage change is **fixing CI and disabling admin-override merges**. Everything else is downhill from there.

2. **Surface area is doubling.** Each merged contributor PR adds a feature (vesting reserves, category duration caps, batch verification, two-step ownership transfer). The README has not kept pace, the threat model has not kept pace, and the test layout is fragmenting. **Either ship a v0.2.0 cut with a frozen feature surface and harden it, or formally split the codebase into a stable "core" and an experimental "extensions" set.** Right now the project advertises a community-validated launchpad but ships an admin-heavy multi-feature platform.
